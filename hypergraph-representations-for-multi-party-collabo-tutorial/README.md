# Hypergraph Representations for Multi-Party Collaborative AI Workflows

A practical demonstration of modeling complex multi-agent collaborative AI workflows using RushDB's property graph as a hypergraph.

## What This Tutorial Demonstrates

This project shows how to use RushDB to model **hypergraph representations** for multi-party collaborative AI workflows. In a hypergraph, edges (called hyperedges) can connect any number of nodes — unlike traditional graphs where edges connect exactly two nodes.

### Key Concepts

1. **Agents** — Autonomous AI agents with distinct roles (researcher, coder, reviewer, synthesizer)
2. **Tasks** — Work units that can be assigned to multiple agents simultaneously (true hyperedge behavior)
3. **Projects** — Shared workspaces containing related tasks and agent contributions
4. **Artifacts** — Generated outputs (code, research, reviews) linked to tasks and agents
5. **Relationships** — Typed connections showing collaboration patterns between all entities

### Why Hypergraph Over Traditional Graph?

In collaborative AI workflows:
- A single task may require **multiple agents** to complete it
- An agent may work on **multiple tasks** simultaneously
- A project involves **many agents** with different roles
- Artifacts are **shared knowledge** across multiple participants

Traditional graph databases model 1:1 or 1:N relationships well, but struggle with true N:M participation. RushDB's property graph supports this through its flexible relationship model combined with record-based properties.

## Prerequisites

- Python 3.9+
- A RushDB account (Free tier available at https://rushdb.com)
- `RUSHDB_API_KEY` environment variable

## Setup

1. **Clone the repository and navigate to the project:**
   ```bash
   git clone https://github.com/rush-db/examples.git
   cd hypergraph-representations-for-multi-party-collabo-tutorial
   ```

2. **Create and activate a virtual environment:**
   ```bash
   python -m venv venv
   source venv/bin/activate  # On Windows: venv\Scripts\activate
   ```

3. **Install dependencies:**
   ```bash
   pip install -r requirements.txt
   ```

4. **Configure your RushDB API key:**
   ```bash
   cp .env.example .env
   # Edit .env and add your RUSHDB_API_KEY
   ```

5. **Seed the database with sample workflow data:**
   ```bash
   python seed.py
   ```

   The seed script creates a realistic multi-agent research collaboration scenario with:
   - 5 AI agents with different roles
   - 4 research projects
   - 12 tasks across projects
   - 15+ artifacts generated by agents
   - Hyperedge relationships showing collaboration patterns

   Expected output:
   ```
   Seeding multi-agent collaborative workflow...
   [1/5] Creating agents...
   [2/5] Creating projects...
   [3/5] Creating tasks...
   [4/5] Creating artifacts...
   [5/5] Creating collaboration relationships...
   Seeding complete!
   ```

## RushDB API Reference

This tutorial uses these RushDB SDK methods:

| Method | Purpose |
|--------|---------|
| `db.records.create()` | Create agents, tasks, projects, artifacts |
| `db.records.create_many()` | Bulk create records |
| `db.records.attach()` | Create hyperedge relationships between records |
| `db.records.find()` | Query records with relationship filtering |
| `db.records.find_by_id()` | Fetch specific records by ID |
| `db.transactions.begin()` | Atomic multi-record operations |

## Key Takeaways

1. **Hypergraph Modeling**: RushDB's property graph supports hypergraph patterns by treating relationships as first-class citizens that can connect any two records.

2. **N:M Relationships**: Tasks can have multiple assigned agents, and agents can work on multiple tasks — this N:M pattern is naturally modeled.

3. **Relationship Traversal**: Use `where` clauses with labels to filter records by related record properties.

4. **Shared Context**: Projects act as central nodes connecting all participants and artifacts in a collaboration.

5. **Atomic Operations**: Use transactions to ensure hyperedge relationships are created consistently.
